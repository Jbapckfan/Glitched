# Glitched iOS Code Review

I reviewed every Swift file in the project, including app/bootstrap, core systems, device-integration managers, UI, character/control code, and all level scenes. I also attempted a build sanity check; it was inconclusive because `xcodebuild` failed in asset compilation due unavailable iPhone simulator runtimes, so the review below is source-based rather than fully compile-validated.

**Findings**
- **Critical:** World progression breaks at the World 1 -> World 2 boundary. `[ProgressManager.swift](/Users/tars/Projects/Glitched/Glitched/Core/ProgressManager.swift#L44)` only unlocks levels in `highestWorld` up to `highestLevelIndex + 1`, so finishing World 1 does not unlock World 2 Level 11 in a menu-driven flow. `[Level10_TimeTravel.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level10_TimeTravel.swift#L1080)` compounds this by transitioning to `LevelID(world: .world2, index: 1)`, but `[LevelFactory.swift](/Users/tars/Projects/Glitched/Glitched/UI/LevelFactory.swift#L35)` defines World 2 starting at index `11`. That path falls through the factory default and is gameplay-breaking.
- **Critical:** `[VoiceCommandManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/VoiceCommandManager.swift#L48)` posts `.voiceCommandMicDenied`, but `[GameInputEvent.swift](/Users/tars/Projects/Glitched/Glitched/Core/GameInputEvent.swift#L4)` does not define that case. This looks like a compile-time source defect.
- **Critical:** Level 18 wires the wrong device mechanic. `[Level18_AppSwitcher.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level18_AppSwitcher.swift#L29)` configures `.appBackgrounding`, which activates `[BackgroundTimeManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/BackgroundTimeManager.swift#L4)`, not `[AppSwitcherManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/AppSwitcherManager.swift#L6)`. The scene listens for `.appSwitcherPeeked`, but the producer for that event is never enabled.
- **Critical:** Level 20 has the same wiring problem. `[Level20_MetaFinale.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level20_MetaFinale.swift#L81)` configures `.appBackgrounding`, but the reinstall mechanic lives in `[ReinstallManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/ReinstallManager.swift#L6)` under `.appDeletion`. The scene waits for `.appReinstallDetected` at `[Level20_MetaFinale.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level20_MetaFinale.swift#L600)`, so the finale’s core gimmick is not actually activated. `ReinstallManager` also calculates `localFlag` but never uses it at `[ReinstallManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/ReinstallManager.swift#L52)`.
- **High:** Level 19 still depends on an unsupported mechanic. `[Level19_FaceID.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level19_FaceID.swift#L32)` configures `[.faceID, .proximity]`, but `[DeviceManagerCoordinator.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/DeviceManagerCoordinator.swift#L22)` has no proximity manager. The scene comment says this was fixed to use real Face ID, but the code still advertises a sensor path that does not exist.
- **High:** Several device integrations are puzzle-unsafe heuristics rather than robust signals. `[FocusModeManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/FocusModeManager.swift#L21)` infers Focus mode from notification settings, which conflates permanent permission state with temporary Focus state. `[NetworkManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/NetworkManager.swift#L44)` infers airplane mode from “no Wi‑Fi, no cellular, no path,” which will false-positive on unrelated offline conditions.
- **High:** Scene transitions are implemented in two places. `[GameRootView.swift](/Users/tars/Projects/Glitched/Glitched/App/GameRootView.swift#L75)` re-presents scenes when `GameState.currentLevelID` changes, while many scenes also call `view.presentScene(...)` directly, for example `[Level1_Header.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level1_Header.swift#L573)`, `[Level11_Notification.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level11_Notification.swift#L674)`, and `[Level30_CreditsFinale.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level30_CreditsFinale.swift#L522)`. That split ownership makes lifecycle behavior harder to reason about and risks duplicate or racing transitions.
- **High:** Architecture is coherent at the folder/file level, but runtime ownership is too singleton-heavy. `[GameState.swift](/Users/tars/Projects/Glitched/Glitched/Core/GameState.swift#L22)`, `[InputEventBus.swift](/Users/tars/Projects/Glitched/Glitched/Core/InputEventBus.swift#L5)`, `[DeviceManagerCoordinator.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/DeviceManagerCoordinator.swift#L12)`, and `[BaseLevelScene.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/BaseLevelScene.swift#L4)` concentrate state into globals that scenes mutate directly. This works for a solo project, but it makes testing, determinism, and scene isolation weak.
- **Medium:** Performance and battery usage are the main non-functional risk. `[BrightnessManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/BrightnessManager.swift#L15)` polls with `CADisplayLink`; `[ClipboardManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/ClipboardManager.swift#L18)`, `[BatteryLevelManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/BatteryLevelManager.swift#L15)`, `[StorageSpaceManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/StorageSpaceManager.swift#L16)`, and `[TimeOfDayManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/TimeOfDayManager.swift#L15)` all poll on timers. `[AudioManager.swift](/Users/tars/Projects/Glitched/Glitched/Core/AudioManager.swift#L294)` allocates and detaches a new `AVAudioPlayerNode` per effect. `[ParticleFactory.swift](/Users/tars/Projects/Glitched/Glitched/Core/ParticleFactory.swift#L333)` recursively respawns digital-rain columns. None of these is fatal alone, but together they create avoidable churn.
- **Medium:** UI/global-state side effects leak outside individual levels. `[Level8_DarkMode.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level8_DarkMode.swift#L42)` flips the app-wide `forceDarkMode` preference on entry and restores it in `[Level8_DarkMode.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level8_DarkMode.swift#L1180)`. It is clever, but it means one level mutates a global presentation preference that affects the rest of the app.
- **Medium:** Error handling is inconsistent. `[Level20_MetaFinale.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level20_MetaFinale.swift#L7)` uses a local `KeychainHelper` but ignores `SecItemAdd` status. `[StorageSpaceManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/StorageSpaceManager.swift#L39)` and other managers rely heavily on `try?` and silent failure. Permission-denied paths are sometimes good, as in `[AuthenticationManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/AuthenticationManager.swift#L43)`, but that rigor is not consistent across managers.
- **Medium:** The AirDrop/share mechanic is internally inconsistent. `[Level28_AirDrop.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level28_AirDrop.swift#L4)` explicitly says the mechanic is “sharing,” not specifically AirDrop, and uses its own share sheet/code path at `[Level28_AirDrop.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level28_AirDrop.swift#L290)`. Meanwhile `[AirDropManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/AirDropManager.swift#L4)` generates a separate expected code and activity controller. The scene and manager are conceptually overlapping rather than cleanly integrated.
- **Low:** There are still unnecessary force unwraps in gameplay code, for example `[Level22_BatteryPercent.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level22_BatteryPercent.swift#L390)`, `[Level25_TimeOfDay.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level25_TimeOfDay.swift#L568)`, and `[Level28_AirDrop.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level28_AirDrop.swift#L541)`. They are easy to remove and would tighten crash resistance.
- **Low:** `[KeyboardState.swift](/Users/tars/Projects/Glitched/Glitched/Input/KeyboardState.swift#L19)` adds a notification observer and never removes it. Because it is a singleton this is unlikely to leak in production, but it is still sloppy lifecycle management.
- **Low:** `[InputEventBus.swift](/Users/tars/Projects/Glitched/Glitched/Core/InputEventBus.swift#L11)` caches “last event by type,” but I found no consumer of `lastEvent(forKey:)` and no caller of `clearLastEvents()`. That replay mechanism is currently dead complexity and a future stale-state hazard if it is only half-adopted.

**Category Assessment**
- **Architecture & code organization:** The project is well-organized physically: one file per level, clear folders, and a readable separation between `Core`, `DeviceIntegration`, `UI`, and `Scenes`. The weak point is logical ownership: many scene files are very large and directly own progression, transitions, device-manager configuration, and level-specific UI, so the architecture scales in file count better than it scales in change safety.
- **SpriteKit usage patterns:** The SpriteKit work is strong stylistically. `[BaseLevelScene.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/BaseLevelScene.swift#L26)` gives every scene a common entry/update/input contract, and the project makes good use of `SKShapeNode`, lightweight physics bodies, and a coherent monochrome visual language. The downside is repeated helper code across scenes and too much transition/lifecycle logic living inside scene subclasses instead of a single coordinator.
- **Device integration quality:** Best-in-class parts are `[MotionManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/MotionManager.swift#L21)` and `[AuthenticationManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/AuthenticationManager.swift#L43)`, which use real platform APIs and contain reasonable fallbacks. The weaker half of the system is the proxy-based integrations and miswired scene configs, which are currently the biggest source of correctness risk.
- **Memory management & performance:** There is good routine use of `[weak self]`, and scene subscriptions are cleared in `[BaseLevelScene.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/BaseLevelScene.swift#L121)`. The main concerns are polling frequency, one-node-per-sound allocation, timer-heavy managers, and long-lived recursive effects. I would profile scene node counts, audio allocations, and background energy before shipping.
- **Game state management:** `[GameState.swift](/Users/tars/Projects/Glitched/Glitched/Core/GameState.swift#L22)` is simple and readable, but progression, UI state, scene transitions, and persistent flags are spread across `GameState`, `ProgressManager`, `UserDefaults`, Keychain, and iCloud KVS. The model works until cross-level edge cases appear, and those edge cases are already visible in world progression and finale/reinstall flows.
- **UI/UX code quality:** The UI code is thematically disciplined and intentionally authored rather than templated. The biggest UX risks are levels that alter global app settings, levels whose real mechanic differs from the player-facing fiction, and a few puzzle states that rely on ambiguous OS signals.
- **Error handling & edge cases:** Permission denial and unavailable hardware are handled well in some places, but too many APIs silently fail. The review found missing compile coverage, unsupported mechanics, dead flags, and unlock-edge bugs; those are exactly the kinds of issues stronger negative-path testing would catch.
- **iOS best practices:** Mixed. Good points include `@MainActor` on the event bus, fresh `LAContext` usage, and use of platform frameworks rather than custom hacks where possible. Weak points include direct `UIApplication`/root-view-controller access from scenes, global singleton coordination, silent Keychain failures, and timer/polling over event-driven APIs.
- **Security considerations:** The app has a relatively small true security surface. The main issues are correctness and spoofability, not classic exploitability: Keychain writes ignore status, reinstall detection relies on state that can drift or be manipulated, and clipboard/share integrations should stay narrowly scoped to active gameplay. There is no obvious secret-management disaster, but the device/meta levels should be explicit that some “secure” mechanics are theatrical rather than authoritative.
- **Level design patterns & consistency:** This is one of the project’s strengths. World themes are distinct, file naming is consistent, and the black/white glitch aesthetic holds across the entire game. The main inconsistency is mechanical truth: by the later worlds, some levels are no longer about the real OS feature named in the title, which makes the design feel less precise than the presentation.

**Areas of Excellence**
- `[BitCharacter.swift](/Users/tars/Projects/Glitched/Glitched/Characters/BitCharacter.swift#L50)` is unusually polished for code-drawn character art. The astronaut silhouette, animation hooks, and physics proportions are thoughtful.
- `[PlayerController.swift](/Users/tars/Projects/Glitched/Glitched/Characters/PlayerController.swift#L125)` handles touch-space vs camera-space correctly and cleanly, which is easy to get wrong in scrolling SpriteKit scenes.
- `[NotificationManager.swift](/Users/tars/Projects/Glitched/Glitched/DeviceIntegration/NotificationManager.swift#L39)` does a good job scoping notification cleanup to game-owned identifiers instead of wiping unrelated pending notifications.
- `[Level29_TheLie.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/Level29_TheLie.swift#L284)` is a good example of extending common systems surgically; `PlayerController.worldWidth` is a small but effective affordance for special geometry.
- `[BaseLevelScene.swift](/Users/tars/Projects/Glitched/Glitched/Scenes/BaseLevelScene.swift#L171)` gives success/failure flow, juice hooks, and event subscription a common base, which is a solid foundation for a scene-heavy game.

**Bottom Line**
Glitched is creatively strong and visually coherent. The strongest code is the handcrafted SpriteKit presentation and the better-integrated device managers; the weakest code is the coordination layer between levels, progression, and device mechanics. If you only fix a few things first, I would prioritize the World 1 -> World 2 progression bug, the undefined voice-command event, the miswired App Switcher/Meta Finale levels, and the proxy mechanics whose false positives can invalidate puzzle logic.

Residual gap: I could not complete a full build in this environment because asset compilation required simulator runtimes that were not installed, and I did not run runtime profiling on-device.
