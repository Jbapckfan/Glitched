Level 5 — Charging

The level gives away its central trick before the player experiments: the battery icon label reads "PLUG IN YOUR CHARGER" (line 751) and the identical `hintText()` override repeats "Connect your device to a charger" (line 1212), so the first encounter is a prompt, not a puzzle. iPhone is essentially a charger-activated elevator, while the iPad climb stays within budget—per-tier rise is clamped to ≤85 (line 94), horizontal columns sit 105pt apart (line 460), and 120pt bridge stones keep edge-to-edge gaps under the limit (lines 481–487)—but the iPad branch is gated by the arbitrary constant `size.height > 1000 && size.width > 430` (line 53), and `buildIPadClimb` silently returns when `padTierCount < 4` (line 441), which can strand wide screens with no exit. Stuck players get no real escalation: there is no `notePlayerStruggle()` override, only the inherited ~18–22 s fallback and a single static `hintText()` string, while the t=0 battery label (line 751) already spoils the solution instead of nudging discovery. Code quality slips with the implicitly unwrapped `floor` force-removed at line 992, the dead expression `_ = startPlatformTopY` (line 531), and the unconditional 0.05 s grounded reset in `didEnd` (lines 1137–1142) that clears grounded state even when another ground body is still touching. Worse, `handleExit()` calls `succeedLevel()` and then schedules its own `transitionToNextLevel()` (lines 1165–1174), risking a duplicate transition if the base success path already handles one. One concrete fix: replace the magic wide-canvas check with an aspect-ratio/form-factor helper, add an explicit assertion or fallback layout for `padTierCount < 4`, and implement a level-specific `notePlayerStruggle()` that first pulses the battery icon and only later reveals the plug hint.
